# 实验助手项目分析建议报告

## 1. 分析范围与依据
- 文档依据：`docs/_converted/案例正文.md`、`docs/_converted/面向程序员的使用步骤.md`、`docs/_converted/实验助手项目部署说明.md`
- 代码依据：`exphlp`（Java 后端多模块）与 `exphlp-front`（Vue2 前端）
- 目标问题：现状不足是什么、如何完成“构建 Python 子系统”（含可选重构路径）

## 2. 目标架构与现实实现的差距
从文档看，项目目标是：算法可插拔、执行计划可观测、结果保存低阻塞、故障可恢复、支持邮件/微信通知与容器化部署。  
当前实现可用，但存在“能跑但不稳、不易扩展”的典型特征：

- 执行内核仍是同步串行调用，失败处理粗糙，难以满足“高可靠+低阻塞”。
- 算法接口停留在 Java `IAlgRun` 语义，跨语言（尤其 Python）未形成稳定契约。
- 配置、认证、安全、部署自动化明显偏弱，生产可运维性不足。

## 3. 关键问题清单（按优先级）

### P0（必须先处理）
1. 执行状态语义不一致  
文档定义“异常结束/正常结束”，代码存在状态映射混乱，成功/失败闭环不完整。

2. 执行链路同步阻塞  
计划执行通过 HTTP 串行触发算法服务，任何节点抖动都会放大为整计划失败。

3. 配置与密钥硬编码  
Mongo/Rabbit/Nacos 地址、账号口令与固定 IP 在代码/部署文档中长期裸露，迁移和安全风险高。

4. 前后端认证体系未真正打通  
前端存在固定 token 与 mock 流程，后端权限边界不清晰。

### P1（应尽快处理）
1. 数据一致性风险  
执行计划字段命名与更新逻辑存在不一致，易导致查询/更新偏差。

2. 通知能力未形成闭环  
文档强调邮件/微信通知，但代码中相关能力可用性与可靠性不足。

3. 工程治理不足  
构建产物与 IDE 文件混入仓库，测试覆盖薄弱，回归风险高。

## 4. “Python 子系统”建设建议

## 4.1 建议路线
优先采用“兼容接入 + 分层重构”的两阶段策略，而非一次性重写。

### 阶段 A：兼容接入（2-4 周）
- 新建 `python-subsystem`（建议 FastAPI）。
- 保持与现有执行器兼容：提供 `POST /myAlg/` 与健康检查 `GET /healthz`。
- 统一输入输出模型（`AlgRunCtx`/`EachResult[]`）的 JSON Schema，避免语言差异导致协议漂移。
- 在算法库中扩展元数据：`runtime=python|java`、`endpoint`、`timeoutMs`、`retries`。
- 先跑通“单算法、单实例、单次运行”最小闭环。

### 阶段 B：执行内核重构（4-8 周）
- 将“执行计划触发”从同步 HTTP 改为异步任务（RabbitMQ/Celery 或 Java 任务队列均可）。
- 执行状态机标准化：`PENDING/RUNNING/SUCCEEDED/FAILED/CANCELLED`，并与前端展示一致。
- 结果保存改为分批写入或流式上报，降低内存峰值与长事务风险。
- 通知改为独立通知通道（事件驱动），避免耦合在主执行路径。

## 4.2 Python 子系统最小技术方案
- Web：FastAPI + Uvicorn
- 校验：Pydantic（严格校验参数与结果类型）
- 任务执行：本地线程池起步，后续迁移 Celery
- 观测：结构化日志（JSON）+ traceId（从执行计划透传）
- 容器：独立 Dockerfile，加入 `docker-compose`

## 5. 是否建议整体重构
建议“分层重构，不建议推倒重来”。

- 推倒重来风险：业务知识流失、交付中断、测试债务迁移失败。
- 分层重构收益：可在保持业务连续的前提下，逐步替换执行内核与接口契约。

## 6. 落地里程碑与验收标准

### M1（2 周）
- Python 服务容器化上线（开发环境）
- Java 可调 Python `/myAlg/`
- 单计划执行成功率 > 95%

### M2（4-6 周）
- 异步执行链路上线（至少支持重试与超时）
- 执行状态、错误码、日志字段统一
- 前端可查看实时执行状态与失败原因

### M3（8-10 周）
- 通知闭环（邮件/微信至少一种稳定可用）
- 配置中心化（环境变量/配置文件分层）
- 回归测试覆盖关键路径（计划创建、执行、结果查询、失败恢复）

## 7. 结论
项目当前最主要矛盾不是“有没有 Python”，而是“执行内核和工程基线是否支撑多语言扩展”。  
先完成协议标准化与兼容接入，再演进到异步执行和状态机治理，能够在最小风险下完成“构建 Python 子系统”并显著提升系统稳定性与可维护性。
