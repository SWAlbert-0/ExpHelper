# 维护手册（Version3）

本文档面向维护者，目标是把常见故障处理标准化，减少重复排查。

## 1. 推荐日常命令

```powershell
# 一键引导（基础容器 + 算法容器 + 健康检查）
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action bootstrap

# 一键发布 webapp/front（强制刷新）
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action deploy

# 运行时体检（容器 + healthz + Nacos服务实例）
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action check

# 启动/检查算法服务（Docker后台常驻，终端关闭不影响）
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action alg-up -Mode offline
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action alg-check

# 一键门禁检查（配置清单 + 单测 + 构建 + 核心e2e + 运行时体检）
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action gate -SkipSmoke

# 仅代码门禁（无 Docker 权限时）
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action gate -SkipSmoke -SkipRuntime
```

执行 `ops.ps1 -Action check` 后，重点关注输出关键词：
- `status=ok, version=..., buildTime=...`
- `service=<serviceName>, total=<n>, healthy=<n>`
- `Runtime readiness check completed.`

### 1.1 关键提醒：后端改动必须更新 webapp 容器

- 只重启前端容器（`exphlp_front`）不会使后端接口变更生效。
- 只执行 `docker restart c_exphlp_webapp` 也不会加载新代码（镜像/JAR 未更新）。
- 后端（Java）有任何改动时，必须至少执行以下之一：
  1. `powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action deploy`
  2. 手动两步：
     - `cd exphlp && mvn -q -DskipTests package`
     - `docker compose --env-file docker/.env -f docker/docker-compose.yml up -d --build --force-recreate webapp`

校验标准：
- `http://localhost:8080/api/auth/healthz` 中 `buildTime` 与本次发布时间一致；
- 页面中对应接口行为已更新，不再出现旧版本表现。
- 脚本最后会输出带参数 URL：`http://localhost:8086/?v=<ts>`，用于穿透浏览器缓存。

### 1.2 配置来源规则（单一真实来源）

- 运行时配置统一从 `docker/.env` 注入（包括邮件、初始化账号、算法调用超时、头像目录等）。
- `docker-compose*.yml` 只引用环境变量，不作为业务配置主存储。
- `application.yml` 仅放默认值，最终以 `docker/.env` 覆盖结果为准。

### 1.3 一键脚本参数（按需）

- 所有运维动作统一通过 `scripts/ops.ps1` 执行，不再保留旧入口脚本。

## 2. 故障定位树

### 2.1 页面提示“系统接口404异常”
1. 先看 `http://localhost:8080/api/auth/healthz` 是否可访问。
2. 确认 `buildTime` 是否是本次重建时间。
3. 若不是，执行：
   - `powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action deploy`
4. 再次验证 `healthz` 后重试页面操作。
5. 若仅前端变化生效而后端未生效，优先判定为 `webapp` 未更新，按 1.1 执行强制重建。

### 2.2 执行前检查失败（Nacos无可用实例）
1. 检查算法服务名与 Nacos 注册名是否一致。
2. 打开 Nacos 服务列表查看健康实例数是否 > 0。
3. 若服务未启动，优先执行：
   - `powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action alg-up -Mode offline`
4. 执行状态核验：
   - `powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action alg-check`
5. 若提示 `docker daemon 不可达` 或 `Access is denied`，请先确认 Docker Desktop 运行中，并使用有 Docker 权限的终端重试。
6. 若提示 `failed to fetch anonymous token`（Docker Hub 网络超时），使用 `scripts/ops.ps1 -Action alg-up -Mode offline`：脚本会先本机打包算法 JAR，再构建运行镜像，避免依赖 `maven` 基础镜像拉取。
7. 新机器/服务器建议直接执行：
   - `powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action bootstrap`

### 2.3 算法调用超时（Read timed out）
1. 查看执行日志是否包含 `reasonCode=ALG_CALL_TIMEOUT`。
2. 若存在，说明算法服务可达但本次参数计算量超出超时阈值。
3. 优先降低参数规模（如 `populationSize`、`maxGenerations`、`runNum`）后重试。
4. 若业务必须大规模运行，提高 `ALG_CALL_READ_TIMEOUT_MS`（默认 `60000`）并重建 `webapp`。

### 2.4 邮件通知未发送
1. 在“执行日志 -> 通知记录”查看 `reasonCode`。
2. 常见错误码与处理：
   - `MAIL_CONFIG_INVALID`：SMTP主机/端口未配置。
   - `MAIL_FROM_EMPTY`：发件人地址为空。
   - `MAIL_FROM_INVALID`：发件地址格式非法。
   - `MAIL_TENCENT_CONFIG_INVALID`：腾讯SES配置缺失。
3. 修复配置后在通知记录中执行补发。

## 3. 关键日志关键词

- 计划执行：`PLAN_START` / `PLAN_DONE` / `PLAN_FAIL`
- 算法调用：`ALG_START` / `ALG_CALL` / `ALG_DONE`
- 重试信息：`RETRY`
- 邮件通知：`MAIL_NOTIFY` / `MAIL_SEND_FAILED`
- 结果接口异常：`RESULT_ENDPOINT_404`

## 4. 维护原则

1. 先复现并留证据（页面、API响应、日志）。
2. 前后端跨层问题必须同步修复。
3. 先通过脚本验证，再做人工点测。
4. 任何配置变更必须同步更新 `docs/dev/配置清单.md`。

## 5. 版本4过渡规则（多语言算法）

1. 算法库新增 `runtimeType` 字段（`java/python`），历史数据默认回填为 `java`。
2. JSON 导入支持 `runtimeType/runtime/language/lang` 字段，无法识别时回退为 `java` 并给出告警。
3. 计划执行阶段仍按 `serviceName` 统一调用，不按语言分叉调用路径。

## 6. 源码上传构建（V4 Batch2）

### 6.1 接口与流程

- 上传源码包：`POST /api/AlgController/uploadSource`（`algId` + `file(zip)`）
- 触发构建启动：`POST /api/AlgController/buildAndStart?taskId=...`
- 查询状态：`GET /api/AlgController/buildStatus?taskId=...`
- 查询日志：`GET /api/AlgController/buildLogs?taskId=...&tail=300`

执行链路：
1. 前端上传 zip（要求根目录包含 `exphlp-alg.json`）；
2. 后端落盘到 `alg.build.work-dir`；
3. 后端按运行环境自动调用：
   - Windows：`scripts/tasks/build-uploaded-alg.ps1`
   - Linux：`scripts/tasks/build-uploaded-alg.sh`
   完成解压、镜像构建、容器启动和 Nacos 注册检查；
4. 前端轮询状态和日志。

### 6.2 运行前提（必须满足）

该能力依赖后端进程可直接调用系统 Docker 引擎。因此生产建议：
- `webapp` 以主机进程方式运行；或
- `webapp` 容器内提供 `docker cli` 且挂载 Docker Socket（`/var/run/docker.sock`）。

若不满足上述前提，构建任务会失败，应退回使用：
- `powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action alg-up -Mode offline`

### 6.3 构建成功但任务仍失败（Nacos 注册超时）

典型现象：
- `buildStatus` 最终为 `FAILED`；
- 构建日志里可见 `docker build` 和 `docker run` 均成功；
- 末尾报错 `wait nacos registration timeout`。

常见根因：
1. 后端容器内开启了 `HTTP_PROXY/HTTPS_PROXY`，Nacos 查询被错误代理；
2. Nacos 检查地址使用不当（容器网络下应优先 `nacos:8848`）。

本项目修复策略（已落地）：
- `scripts/tasks/build-uploaded-alg.sh` 在 Nacos 检查时使用 `curl --noproxy '*'`；
- 地址选择规则为：
  - 优先 `NACOS_SERVER_ADDR`；
  - 容器网络下默认 `nacos:8848`；
  - 非容器网络回退 `host.docker.internal:8848`。

复验命令：
```powershell
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action deploy
```
然后执行“源码上传 -> 构建”，日志中应出现：
- `[OK] service registered to nacos, healthy=...`
