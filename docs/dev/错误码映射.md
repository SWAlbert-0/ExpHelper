# 错误码映射

> 文档状态：长期维护  
> 用途：统一后端错误码、任务内部错误码与前端提示语义，作为联调和排障基准。  
> 代码基线：`exphlp/api/webApp/src/main/java/fjnu/edu/auth/ErrorCode.java`  
> 历史材料：`docs/archive/dev/`

## 1. 统一响应结构

后端接口统一返回以下字段：

- `code`：HTTP语义对应状态码（200/400/401/404/409/500）
- `msg` / `message`：提示信息
- `data`：业务数据
- `errorCode`：业务错误码（失败时）
- `traceId`：请求链路标识
- `path`：请求路径
- `timestamp`：毫秒时间戳

## 2. 后端 ErrorCode 全量映射（与代码同步）

## 3. 构建任务内部错误码（Alg Build Task）

说明：以下错误码不在 `ErrorCode.java` 枚举中，由算法构建任务状态机内部维护，主要通过以下接口返回：

- `POST /api/AlgController/uploadSource`
- `POST /api/AlgController/buildAndStart`
- `GET /api/AlgController/buildStatus`
- `GET /api/AlgController/buildLogs`

| internalErrorCode | 场景 | 建议处理 |
| --- | --- | --- |
| `ALG_BUILD_FAILED` | 源码构建、镜像启动或 Nacos 注册检查失败 | 查看 `buildLogs` 末尾日志并按提示修复后重试 |

任务状态字段：

- `status`：`PENDING` / `RUNNING` / `SUCCESS` / `FAILED`
- `phase`：`VALIDATE_PASSED` / `BUILD_AND_START` / `CHECK_PASSED` / `FAILED`
- `errorCode`：任务内部错误码（失败时）
- `errorMessage`：失败原因摘要
- `fixHints[]`：修复建议（用于前端“下一步如何处理”提示）

## 4. 执行日志阶段码（通知相关）

| stage | reasonCode | 含义 |

| `MAIL_NOTIFY` | `MAIL_NO_RECEIVERS` | 计划未配置通知用户 |
| `MAIL_NOTIFY` | `MAIL_OUTBOX_CREATED` | 异步通知任务已入队 |
| `MAIL_NOTIFY` | `MAIL_QUIET_HOURS_DELAYED` | 命中静默时段，通知延迟发送 |
| `MAIL_NOTIFY` | `MAIL_MANUAL_RESEND_CREATED` | 手动补发任务已创建 |
| `MAIL_NOTIFY` | `MAIL_USER_DISABLED` | 用户关闭计划结束通知 |
| `MAIL_NOTIFY` | `MAIL_CONFIG_INVALID` | SMTP 配置缺失（host/port） |
| `MAIL_NOTIFY` | `MAIL_FROM_EMPTY` | 发件邮箱为空 |
| `MAIL_NOTIFY` | `MAIL_FROM_INVALID` | 发件邮箱格式非法 |

| `MAIL_NOTIFY` | `NOTIFY_CHANNEL_UNSUPPORTED` | 当前通知通道暂不支持发送 |
| `WECHAT_NOTIFY` | `WECHAT_NOT_IMPLEMENTED` | 微信通知为扩展预留，当前版本未启用实际发送 |

| `MAIL_NOTIFY` | `PLAN_EXCEPTION_EVENT_CREATED` | 执行异常事件通知任务已入队 |

## 5. 前端提示映射策略

前端在 `exphlp-front/src/utils/errorCode.js` 按以下优先级展示：

1. `errorCode` 映射
2. `code` 映射
3. 后端 `msg/message`
4. 默认兜底文案

额外约束：`/admin/*` 请求在前端请求层直接阻断，映射 `LEGACY_ADMIN_DISABLED`。
