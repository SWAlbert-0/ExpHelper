## 4. Python 详细设计与实际接口不一致：`/execute`、`NacosClient`、`AlgorithmFactory`

### 替换位置 A（6.1 整段）
- 定位起始句：`Python服务端基于FastAPI框架构建，采用分层架构设计`
- 定位结束句：`'''发送心跳'''`

### 修改理由
- 文中 `@app.post("/execute")`、`NacosClient`、`AlgorithmFactory`、`AlgorithmExecutor` 等描述属于概念化写法，与当前项目主链路接口和实体不一致。
- 真实链路统一使用 `POST /myAlg/` 与 `GET /myAlg/`。

### 证据路径
- `docs/templates/python-fastapi-nacos/main.py`
- `exphlp/api/clientApi/src/main/java/fjnu/edu/impl/PlanExecuteImpl.java`
- `exphlp/api/webApp/src/main/java/fjnu/edu/controller/ExePlanMgrCtrl.java`

### 建议替换
Python 服务端在本课题中的定位是按平台约定提供计算能力。其核心职责包括接收运行上下文、执行算法计算、返回结构化结果，并响应健康检查请求。  
在接口约定上，平台以 `POST /myAlg/` 作为执行入口，以 `GET /myAlg/` 作为健康检查入口，该约定与执行前检查、服务发现和运行态管理流程保持一致。  
在实现层面，服务内部可根据算法复杂度采用适当的组织方式，本文重点描述其与平台主链路的一致性与可维护性，而不限定单一内部类结构。

### 替换位置 B（6.2 关键段）
- 定位句：`算法适配层的核心接口为AlgorithmExecutor`

### 建议替换
算法适配层的重点是保持统一输入输出约定，而不是限定固定类名。对于新算法接入，关键在于接口契约对齐、服务配置一致、执行前检查通过以及结果可归档。该设计在保证平台稳定性的同时，保留了算法实现侧的演进空间。

## 5. 容器启动路径描述过窄：仅 `entrypoint.sh + ALG_TYPE`

### 替换位置

- `scripts/tasks/build-uploaded-alg.sh`
- `exphlp/api/webApp/src/main/java/fjnu/edu/algruntime/service/impl/AlgBuildTaskServiceImpl.java`

### 建议替换
当前系统中，算法容器主要有两类启动路径：一类用于示例算法服务的统一启动脚本；另一类用于源码上传后的动态构建与启动。两类路径在实现细节上可不同，但均需满足服务命名一致、实例可注册、接口可探测等基本要求，从而保证执行链路稳定运行。

## 6. Nacos 注册机制描述需收敛为“平台统一判定，可多实现路径”

### 替换位置
- 定位句：`服务注册与发现基于Nacos实现。算法服务容器启动时，entrypoint.sh脚本会读取环境变量ALG_SERVICE_NAME和ALG_PORT，并通过Nacos客户端SDK将服务实例注册到Nacos注册中心。`

### 修改理由
- 当前项目允许不同服务采用不同注册实现方式，但平台侧统一按服务名发现与可用实例判定。
- 建议避免写成单一实现方案。

### 证据路径
- `exphlp/api/webApp/src/main/java/fjnu/edu/controller/ExePlanMgrCtrl.java`
- `exphlp/api/clientApi/src/main/java/fjnu/edu/impl/PlanExecuteImpl.java`
- `docs/paper/paper_figures/fig2_nacos_sequence.md`

### 建议替换
服务注册与发现由运行时启动流程与平台调用流程共同保障。算法服务启动后按约定服务名进入注册中心，平台通过服务发现组件获取可用实例，并结合执行前检查判断实例状态。不同算法服务可采用不同的注册实现方式，但平台侧保持统一的服务名解析与可用性判定机制。

## 8. 测试章节中的“固定值表述”建议改为“批次证据表述”
