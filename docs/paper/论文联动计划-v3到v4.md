# 论文联动计划（v3 -> v4）

## 1. 写作主线与范围

- 论文主线：`Python端算法实验子系统`。
- 前置工作定位：当前 Java/Vue 的完善与纠错作为“平台基础能力与联调前置工作”写入论文，不喧宾夺主。
- 目标：形成“可部署、可执行、可观测、可通知、可对比评估”的实验平台闭环。

## 2. 章节建议（面向最终稿）

1. 引言（背景、问题、目标、贡献）
2. 相关技术与平台前置基础（微服务、Nacos、Docker、消息与通知）
3. Python端需求分析（功能/非功能）
4. Python端架构与关键设计（服务注册、调用协议、数据模型）
5. Python端实现（模块实现与关键代码）
6. 测试与结果分析（功能、联调、异常恢复、指标对比）
7. 总结与展望

## 3. UML 图要求（必须落地）

第一批必须完成三类图：

- 用例图（系统角色与功能边界）
- 类图（核心实体与关系）
- 部署图（容器、服务、Nacos、数据库、消息组件）

后续按功能补充：

- 时序图（计划执行、通知投递、重试链路）
- 活动图（执行向导与失败恢复）

规范：

- 图号按章编号：`图4-1`、`图5-2`。
- 每张图必须在正文中有“引用 + 解释 + 结论”三段说明。
- 图名与图体不跨页，图名居中。

## 4. 指标体系口径（论文与系统保持一致）

当前统一指标集：

- `HV`（越大越好）
- `IGD+`（越小越好）
- `GD`（越小越好）
- `Coverage`（同计划算法两两比较，越大越好）
- `Spread(Δ)`（经典Δ定义，越小越好）
- `Spacing`（越小越好）

用途归类：

- 逼近能力：`IGD+`、`GD`
- 多样性/分布：`HV`、`Spread(Δ)`、`Spacing`
- 算法间覆盖比较：`Coverage`

## 5. 功能完成即论文同步（执行规则）

每完成一个 Python 功能，必须同步更新论文，最少包含：

1. 设计说明：新增能力、边界、输入输出。
2. 实现说明：关键实现点（接口/数据结构/异常处理）。
3. 测试证据：命令、结果、关键日志或截图。
4. 图表更新：至少一张相关图（UML 或执行结果图）。

未完成上述四项，不认定该功能“阶段完成”。

## 6. 版本衔接规则

- `版本3`：完成当前指标体系升级与现有平台收口。
- `版本4`：进入 Python 端功能迭代与论文持续联动更新。
- 分支切换时保持文档与代码同版本提交，避免“代码更新但论文滞后”。
