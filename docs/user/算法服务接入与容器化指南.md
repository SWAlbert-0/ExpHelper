# 算法服务接入与容器化指南

本文档面向“有自定义算法服务”的用户，聚焦接入与执行联通。

基础部署与页面操作请先看：`快速上手-从部署到执行.md`。

## 0. 推荐起点：使用官方模板

优先从模板复制再改造，避免接口和目录不兼容：

- Java 模板：`docs/templates/java-springboot-nacos`
- Python 模板：`docs/templates/python-fastapi-nacos`

两套模板均已包含：
- `POST /myAlg/` 执行接口
- `GET /myAlg/` 健康接口
- `exphlp-alg.json` 上传构建元数据

案例参考（可直接做端到端演练）：
- `docs/cases/moo-python-nsga2-zdt1`
- `docs/cases/moo-python-moead-lsmop`

## 1. 接入目标

你的算法服务需要满足：

1. 提供接口：`POST /myAlg/`
2. 可在 Nacos 注册
3. 服务名全链路一致：
   - 算法服务注册名
   - 算法库中的 `serviceName`
   - 执行计划调用的服务名

## 2. 平台支持的接入方式

### 2.1 方式A：前端源码上传构建（推荐）

页面路径：`算法库管理 -> 源码`

上传 zip 要求：
- 根目录包含 `exphlp-alg.json`

最小示例：

```json
{
  "runtimeType": "java",
  "port": 18082
}
```

字段说明：
- `runtimeType`：`java` 或 `python`
- `port`：算法服务容器内监听端口
- `entry`：Python 可选，默认 `main:app`
- `serviceName`：建议必填并与算法库 `serviceName` 完全一致（可在上传阶段做契约一致性检查）

上传后流程：
1. 点击“构建并启动”
2. 查看阶段状态（`VALIDATE_PASSED -> BUILD_AND_START -> CHECK_PASSED`）
3. 等待任务状态变为 `SUCCESS`
3. 在 Nacos 确认服务有健康实例
4. 再去执行计划

源码弹窗新增运行态信息与运维动作：
- 默认每 10 秒自动刷新一次，可手动点击“刷新”。
- 可查看最近一次上传和构建信息（状态、阶段、错误码、时间、日志）。
- 对于管理员和算法上传者，可执行：`下线`、`上线`、`重启`、`清理旧资源`。
- 若显示“仅可查看运行信息”，说明当前账号不是管理员且非该算法上传者。

若上传失败并返回 `ALG_BUILD_CONTRACT_INVALID`，优先按页面中的“修复建议”处理：
- 对齐算法库 `runtimeType` 与源码包 `exphlp-alg.json.runtimeType`
- 对齐算法库 `serviceName` 与源码包 `exphlp-alg.json.serviceName`
- Java补齐 `pom.xml`，Python补齐 `requirements.txt` 或 `pyproject.toml`

### 2.2 方式B：算法服务容器统一启动

```powershell
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action alg-up -Mode offline
```

检查：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action alg-check
```

## 3. 算法库入库字段规范

必填：
- `algName`
- `serviceName`
- `runtimeType`

JSON 导入兼容字段：
- `runtimeType` / `runtime` / `language` / `lang`
- 未识别时默认按 `java` 处理

## 4. 执行前检查标准

计划执行前，至少满足：

1. 算法服务在 Nacos 有健康实例（`healthy > 0`）
2. 计划中算法 `serviceName` 与 Nacos 服务名一致
3. 问题实例、算法参数填写完整

补充说明：
- 问题实例中的 `machineName/machineIp/dirName` 主要用于记录实验环境上下文与追溯；
- 当前版本不按该字段进行调度路由。

## 5. 接入验收最小清单

1. 算法入库成功
2. Nacos 可见目标服务实例
3. 执行计划“执行检查”通过
4. 计划执行状态收敛到“正常结束”或“异常结束（含原因）”
5. 执行日志中存在 `ALG_START/ALG_CALL/ALG_DONE`（或失败阶段）

## 6. 常见失败与处理

详见：`常见问题与排障.md`

重点关注：
- `Nacos中无可用实例`
- `wait nacos registration timeout`
- `ALG_CALL_TIMEOUT`
- `RESULT_ENDPOINT_404`

补充说明（Python 算法常见）：
- 若容器状态是 `Up` 但 Nacos 健康实例为 0，优先检查容器内代理变量（`HTTP_PROXY/HTTPS_PROXY`）是否干扰了 Nacos 注册。
- 平台最新构建脚本已在上传构建时自动清空代理变量并使用可重试心跳；旧镜像建议在“源码”弹窗再执行一次“构建并启动”完成升级。
