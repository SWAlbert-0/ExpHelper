# 容器说明与职责清单

本文档说明本项目各容器的职责、依赖关系和常见运维检查项。

## 1. 核心业务容器

### 1.1 `c_exphlp_webapp`
- 作用：后端 API 服务（鉴权、计划管理、算法调用、结果计算、通知任务等）。
- 对外端口：`8080`
- 依赖：`mongo`、`rabbitmq`、`nacos`
- 关键检查：
  - `GET /api/auth/healthz`
  - 执行日志中 `PLAN_START/PLAN_DONE/PLAN_FAIL`

### 1.2 `c_exphlp_vue_front`
- 作用：前端页面（管理台 UI）。
- 对外端口：`8086`
- 依赖：`webapp`
- 关键检查：
  - `http://localhost:8086`
  - 页面操作是否与后端行为一致

## 2. 中间件容器

### 2.1 `c_exphlp_mongo`
- 作用：持久化存储（问题实例、算法、计划、执行结果、通知记录等）。
- 对外端口：`27017`
- 关键检查：
  - 容器 `healthy`
  - 读写是否正常

### 2.2 `c_exphlp_rabbitmq`
- 作用：异步通知任务队列（邮件通知分发）。
- 对外端口：`5672`，管理端口 `15672`
- 关键检查：
  - 容器 `healthy`
  - 是否存在堆积消息

### 2.3 `c_exphlp_nacos`
- 作用：服务注册与发现（算法服务注册中心）。
- 对外端口：`8848`、`9848`、`9849`
- 关键检查：
  - `http://localhost:8848/nacos`
  - 服务列表健康实例数是否 > 0

## 3. 算法服务容器（示例）

- `c_nsga2_zdt1_example` -> `nsga2-zdt1-ls`
- `c_moead_lsmop_example` -> `moead-lsmop-ls`
- `c_smpso_lsmop_example` -> `smpso-lsmop-ls`
- `c_spea2_lsmop_example` -> `spea2-lsmop-ls`

作用：
- 实际承接 `/myAlg/` 计算请求。
- 通过 Nacos 供平台动态发现。
- 四个示例容器复用同一镜像 `exphlp-alg-runner:local`，通过 `ALG_TYPE` 区分算法类型。

关键检查：
- 容器状态 `Up`
- Nacos 对应服务名有健康实例
- 执行计划日志无 `Nacos中无可用实例`

## 4. 依赖关系（简化）

1. 前端 -> 后端 API
2. 后端 -> Nacos 发现算法服务
3. 后端 -> 算法服务 `/myAlg/`
4. 后端 -> Mongo 持久化
5. 后端 -> RabbitMQ 投递通知

## 5. 运维常用命令

```powershell
# 核心容器运行就绪
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action check

# 新机器/服务器一键部署
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action bootstrap

# 启动算法服务容器（后台常驻）
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action alg-up -Mode offline

# 检查算法服务容器 + Nacos实例
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action alg-check

# 停止算法服务容器
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action alg-down
```

## 6. 常见状态解释

- `计划状态=正常结束` 但结果为空：检查算法服务返回内容是否含有效指标/Pareto 点。
- `Read timed out`：算法可达但参数规模过大或计算超时。
- `Nacos中无可用实例`：服务未注册、服务名不一致或实例不健康。
