# 快速上手：从 Docker 部署到计划执行

本文面向平台使用者，目标是在最少前置知识下，完成一次完整闭环：

问题实例入库 -> 算法入库 -> 计划执行 -> 日志/结果查看。

## 0. 任务导航（先做什么）

- 首次部署：看第 1~2 节。
- 快速检查环境是否可用：看第 2 节的 `scripts/ops.ps1 -Action check`。
- 做一次完整业务闭环：按第 3~7 节。
- 排查执行失败或通知失败：看第 8 节。
- 需要重建页面与后端：看第 9 节。

## 1. 前提检查

- 已安装 Docker Desktop（Windows/macOS）或 Docker Engine（Linux）
- 本地端口未占用：`8080`、`8086`、`8848`、`9848`、`9849`

## 2. 启动平台（WebApp + Front + 依赖）

在仓库根目录执行（推荐使用一键脚本，自动先打包后端再重建容器）：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action deploy
```

新机器/服务器推荐直接执行一键引导（基础容器 + 算法容器 + 健康检查）：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action bootstrap
```

或手工执行：

```powershell
docker compose --env-file docker/.env -f docker/docker-compose.yml up -d --build mongodb rabbitmq nacos webapp exphlp_front
```

启动后建议先执行运行自检：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action check
```

访问地址：

- 前端：`http://localhost:8086`
- Nacos：`http://localhost:8848/nacos`
- 健康检查：`http://localhost:8080/api/auth/healthz`（可查看 `artifactVersion/buildTime` 判断是否为最新后端）

头像持久化说明（已内置）：

- 当前部署默认启用头像目录持久化卷，容器重建后头像文件不会丢失。
- 关键变量：`AUTH_AVATAR_DIR=/app/data/uploads/avatar`（见 `docker/.env`）。
- 如历史数据出现“数据库有头像路径但文件不存在”，页面会自动回退默认头像，用户可在“个人中心”重新上传。

如需启用“执行结果邮件通知”，请先在 `docker/.env` 增加 SMTP 配置（示例）：

```dotenv
SPRING_MAIL_HOST=smtp.example.com
SPRING_MAIL_PORT=587
SPRING_MAIL_USERNAME=your-account@example.com
SPRING_MAIL_FROM=your-account@example.com
SPRING_MAIL_PASSWORD=your-password
SPRING_MAIL_PROPERTIES_MAIL_SMTP_AUTH=true
SPRING_MAIL_PROPERTIES_MAIL_SMTP_STARTTLS_ENABLE=true
```

可选：启用腾讯SES作为备通道（推荐在生产环境开启）：

```dotenv
NOTIFY_MAIL_PROVIDER_STRATEGY=smtp_primary_ses_fallback
NOTIFY_MAIL_TENCENT_ENABLED=true
NOTIFY_MAIL_TENCENT_ENDPOINT=ses.tencentcloudapi.com
NOTIFY_MAIL_TENCENT_REGION=ap-guangzhou
NOTIFY_MAIL_TENCENT_SECRET_ID=AKIDxxxxxxxx
NOTIFY_MAIL_TENCENT_SECRET_KEY=xxxxxxxx
NOTIFY_MAIL_TENCENT_FROM_EMAIL=notice@example.com
NOTIFY_MAIL_TENCENT_FROM_NAME=expHlp
```

## 3. 启动示例算法服务（容器化）

```powershell
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action alg-up -Mode offline
```

说明：
- 脚本会先在本机执行算法服务 `mvn package`，再启动容器。
- 该流程可规避部分环境下 Docker Hub 拉取 `maven` 构建镜像超时的问题。

校验算法服务是否已注册到 Nacos：

1. 打开 `http://localhost:8848/nacos`
2. 进入“服务管理 -> 服务列表”
3. 确认存在服务名：`nsga2-zdt1-ls`，并且健康实例数大于 0

可选校验命令：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action alg-check
```

## 4. 问题实例入库（网页）

菜单：`问题实例管理` -> `添加`

也可使用：`问题实例管理` -> `JSON导入`，支持拖拽 `.json` 文件。  
导入器会自动将旧字段（`probName/probDesc/probClassName/catalogName/ip`）标准化为当前字段。

建议先用示例字段：

- 实例名称：`zdt1-ls100-caseA-ui`
- 类别：`MOO`
- 其余字段按页面提示填写

重点说明：

- “实例名称”用于后续计划中识别问题实例。
- “类别”用于组织展示，不影响算法计算逻辑。

## 5. 算法入库（网页）

菜单：`算法库管理` -> `添加`

也可使用：`算法库管理` -> `JSON导入`，支持拖拽 `.json` 文件。  
导入器会自动兼容 `algorithmName/service/params` 等历史字段并转换为标准结构。

关键字段必须一致：

- 算法名称：`nsga2-zdt1-ls`（建议与服务名一致）
- 服务名：`nsga2-zdt1-ls`（必须与 Nacos 注册名一致）

参数填写示例：

- `populationSize`（`int`）
- `maxGeneration`（`int`）
- `seed`（`long`）

## 6. 执行计划

菜单：`执行计划管理`

推荐路径：

1. 点击 `执行向导`
2. 选择问题实例
3. 选择算法并配置参数
4. 点击 `执行检查`
5. 通过后点击执行

## 7. 查看日志与结果

- 在计划详情页点击 `执行日志` 可实时查看日志。
- 支持导出日志：`CSV` / `JSON`。
- 在执行结果弹窗查看指标汇总与每次运行结果。
- 执行日志中 `MAIL_NOTIFY` 阶段用于展示邮件通知结果（成功/未发送/失败原因）。
- 执行日志弹窗下半区可查看“通知记录”，对失败通知可单条补发或“补发本批次失败通知”。
- 计划列表中的“结束时间”在未执行/执行中会显示 `--`（表示尚未结束），不会再显示 `1970-01-01 08:00:00`。

### 7.1 个人通知设置（网页）

菜单：`个人中心 -> 通知设置`

可配置项：

- 通知邮箱（优先于个人资料邮箱）
- 是否启用邮件通知
- 是否启用计划结束通知
- 静默时段（命中后自动延迟发送）
- 时区（默认 `Asia/Shanghai`）
- 点击“发送测试邮件”可立即按当前用户通知邮箱发送一封测试信，用于验证配置是否可用

多目标常见指标说明：

- `HV`：超体积（越大越好）
- `IGD+`：反向收敛距离（越小越好）
- `GD`：代际距离（越小越好）
- `Coverage`：覆盖率（同计划算法两两比较，越大越好）
- `Spread(Δ)`：分布均匀性（经典Δ，越小越好）
- `Spacing`：相邻解间距离离散度（越小越好）

Coverage 判读补充：
- 显示 `N/A`：当前没有可比较算法（例如计划中只有一个算法）。
- 显示 `0.000000`：有可比较算法，但当前算法未覆盖对方前沿点；这是合法结果，不代表系统故障。

## 8. 常见问题

### 8.1 提示“Nacos 中无可用实例”

- 核查算法“服务名”是否与 Nacos 注册名一致。
- 核查 Nacos 地址是否可达（容器内应使用 `nacos:8848`）。
- 执行 `powershell -ExecutionPolicy Bypass -File scripts/ops.ps1 -Action alg-check` 确认算法容器与实例健康。

### 8.2 执行结果弹窗 404

- 前端会自动降级展示基础结果。
- 建议重建并重启 `webapp`：

```powershell
docker compose --env-file docker/.env -f docker/docker-compose.yml up -d --build webapp
```

### 8.3 删除后列表未变化

- 点击页面 `刷新` 重新拉取列表。
- 新版本会区分三种状态：
  - 已实际删除
  - 记录已不存在并完成同步
  - 执行中被阻止删除

### 8.4 邮件通知未发送

- 在计划执行日志中查找 `MAIL_NOTIFY` 阶段：
  - `reasonCode=MAIL_NOT_CONFIGURED`：SMTP 未配置；
  - `reasonCode=MAIL_SEND_FAILED`：SMTP 调用失败；
  - `reasonCode=MAIL_USER_NOT_FOUND`：通知用户不存在。
  - `reasonCode=MAIL_CONFIG_INVALID`：`SPRING_MAIL_HOST/PORT` 缺失。
  - `reasonCode=MAIL_FROM_EMPTY`：`SPRING_MAIL_FROM` 和 `SPRING_MAIL_USERNAME` 均为空。
  - `reasonCode=MAIL_FROM_INVALID`：`SPRING_MAIL_FROM` 格式错误。
  - `reasonCode=MAIL_TENCENT_CONFIG_INVALID`：腾讯SES配置缺失或未启用。
  - `reasonCode=MAIL_TENCENT_SEND_FAILED`：腾讯SES调用失败（检查密钥/地域/发信地址）。
  - `reasonCode=MAIL_PROVIDER_FALLBACK_USED`：SMTP失败后已由腾讯SES成功发出。
  - `reasonCode=MAIL_PROVIDER_ALL_FAILED`：SMTP与腾讯SES都失败。
- 确认 `docker/.env` 中 SMTP 配置完整后，重启 `webapp` 容器：

```powershell
docker compose --env-file docker/.env -f docker/docker-compose.yml up -d --build webapp
```

### 8.5 头像显示为默认图

- 如果是首次使用，属于正常（尚未上传头像）。
- 如果之前上传过头像但当前显示默认图，通常是历史文件缺失：
  - 页面会自动回退默认头像，避免破图；
  - 进入 `个人中心` 重新上传头像即可恢复。
