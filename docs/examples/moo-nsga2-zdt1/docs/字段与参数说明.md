# NSGA2-ZDT1 字段与参数说明（重写版）

本说明文档回答两个问题：
1. 每个字段该怎么填、为什么这么填。  
2. 异常时该看哪里、怎么快速定位。

## 1. 关键概念

- 算法名称：仅用于页面展示和人工识别。
- 服务名：平台真实调用的服务标识，必须等于 Nacos 注册名。
- 本案例 Nacos 注册名：`nsga2-zdt1-ls`

建议：
- 算法名称与服务名保持同名（都用 `nsga2-zdt1-ls`），减少排错成本。

## 2. 问题实例字段含义

- 实例名称：唯一可读标识，建议包含问题+规模。
- 类别名称：用于树形分组；同类实例建议统一。
- 目录名称：业务归档路径标签，不参与算法计算。
- 机器名称：来源机器标签，便于排查。
- IP地址：实例来源地址，建议填可达地址。
- 实例描述：记录变量规模、用途、数据背景。

## 3. 算法字段含义

- 算法名称：展示名，建议 `nsga2-zdt1-ls`。
- 服务名：调用名，必须 `nsga2-zdt1-ls`。
- 算法描述：简述算法目的和适用范围。

### 3.1 参数公共字段

- 参数名：算法读取该参数的 key。
- 参数类型：参数元类型（当前支持 `int`、`double`、`long` 等）。
- 参数值：页面输入字符串，算法服务按类型解析。
- 描述：参数业务含义。

### 3.2 本案例参数解释

- `nVars` (`int`)：决策变量个数；示例 `100`。
- `populationSize` (`int`)：种群规模；越大一般越稳但更慢。
- `maxGenerations` (`int`)：进化代数；越大一般更充分但更耗时。
- `crossoverProbability` (`double`)：交叉概率，常用 `0.9`。
- `mutationProbability` (`double`)：变异概率，常用 `0.01`。
- `seed` (`long`)：随机种子，便于复现实验。

## 4. 执行计划字段含义

- 计划名称：建议唯一，便于回溯。
- 问题实例：本次要被求解的数据对象。
- 算法配置：包含算法选择、参数、运行次数。
- 运行次数：同一配置重复执行次数。
- 通知人员：执行结束通知接收者。
- 计划描述：本次实验目标说明。

## 5. 执行状态与策略

- 未执行：已创建未投递。
- 执行中：已受理执行。
- 正常结束：结果生成完成。
- 异常结束：执行失败，可查看异常原因并重试。

重试策略：
- 仅 `异常结束` 可点击 `重新执行`。
- 同一 `planId + algId + algName` 结果按最新执行覆盖展示。

## 6. 执行前连通性检查（新增）

平台在受理执行前会检查：
- 算法配置中的 `serviceName` 是否存在。
- Nacos 中是否有该 `serviceName` 的可用实例。

检查失败时不会进入“执行中”，会直接失败并提示类似：
- `执行前检查失败: 服务[nsga2-zdt1-ls]在Nacos中无可用实例`

## 7. 执行日志说明（实时）

- 入口：执行计划详情 -> `执行日志`
- 刷新：每 2 秒轮询
- 字段：
  - `seq`：该计划内递增序号
  - `ts`：时间戳
  - `level`：`INFO/WARN/ERROR`
  - `stage`：阶段（`PLAN_START`、`ALG_CALL`、`RETRY`、`PLAN_DONE`、`PLAN_FAIL`）
  - `message`：可读详情

## 8. 常见错误与处理

### 8.1 `Service Instance cannot be null`

典型原因：
- 算法服务名不匹配（例如页面写成 `nsga2-zdt1-ls-ui`，实际注册是 `nsga2-zdt1-ls`）。

处理：
1. 在 Nacos 确认注册名。
2. 在算法库把服务名改成一致值。
3. 重新执行计划。

### 8.2 `No servers available for service: xxx`

典型原因：
- 该服务在 Nacos 没有可用实例（未启动、崩溃、网络不可达）。

处理：
1. 重启算法服务；
2. 在 Nacos确认实例健康；
3. 再执行计划。

### 8.3 `Unsupported class file major version 61`

根因：
- Java 17 与旧版 Spring Boot 不兼容。

本案例固定版本：
- Spring Boot `2.7.18`
- Spring Cloud `2021.0.8`
- Spring Cloud Alibaba `2021.0.5.0`

### 8.4 删除提示与数据一致性

删除问题实例/算法时，页面提示可按以下规则解读：

- `删除成功`：
  - 后端已确认删除（`deletedCount > 0`）；
  - 前端会刷新列表，目标行应消失。
- `记录已不存在，列表已同步`：
  - 后端判定该记录已不存在（`noop=true` 且 `verified=true`）；
  - 属于幂等删除场景，不是故障。
- `问题实例已被执行计划引用`：
  - 命中引用保护（`errorCode=PROB_IN_USE`）；
  - 需要先解除执行计划关联，再删除。
- `删除未生效，请刷新后重试`：
  - 后端未确认删除（`noop=true` 且 `verified=false`）；
  - 需检查后端日志、Mongo 记录字段与 ID 类型是否一致。
